# Nabi Markdown Repository Foundation Design

**Status:** Approved for repository bootstrap on 2026-07-18

**Scope:** Documentation, licensing, Build Week evidence, and GitHub setup only

## Objective

Create a public, judge-readable repository for Nabi Markdown before feature
implementation begins. The repository must preserve Jiwon's voice, support a
future paid hosted product, and collect truthful evidence of Codex and GPT-5.6
use throughout OpenAI Build Week.

## Repository contract

- Owner and repository: `jiwonschol/nabimd`
- Visibility: public
- Default branch: `main`
- Bootstrap work branch: `agent/bootstrap-nabi-markdown`
- Code license: AGPL-3.0-or-later
- Product name and future visual identity: outside the code license
- External contributions: not merged until judging ends

The AGPL keeps modified network deployments open. It does not prevent the
project owner from charging for an official hosted service, content, support,
or a separate commercial license. Commercial terms are outside the Build Week
scope.

## Initial files

- `README.md`: canonical founder story, product promise, truthful current
  status, Build Week collaboration evidence, and eventual run/test path
- `LICENSE`: official GNU AGPL v3 text generated by GitHub
- `TRADEMARKS.md`: separates the name and future identity from the code license
- `CONTRIBUTING.md`: prevents ownership ambiguity during the competition
- `.gitignore`: future Vite/Node output and secret exclusions
- `docs/build-log.md`: dated problems, decisions, Codex contribution, evidence
- `docs/submission-checklist.md`: official submission and README audit

## Evidence model

The project uses one primary Codex thread for the majority of core work. Each
meaningful milestone records:

1. the observed problem;
2. Codex's proposal or acceleration;
3. Jiwon's decision;
4. the verification result; and
5. unresolved limitations.

After core functionality is substantially complete, `/feedback` is run in the
primary thread and the Session ID is pasted into Devpost. The public README and
build log explain the workflow without publishing a fabricated history.

## Error and truthfulness policy

- Planned features are labeled as planned until runnable.
- Failed approaches remain in the log when they influenced the result.
- User research and automated testing are never presented as equivalent.
- The Devpost plugin is a helper; Official Rules and FAQ remain authoritative.
- Missing credits, API costs, and tool boundaries are documented honestly.

## Verification

The bootstrap is complete when:

- GitHub reports the repository as public;
- GitHub detects the AGPL license;
- the branch contains only the approved foundation files;
- Markdown links resolve within the repository;
- no secret or unfinished application claim is present; and
- a draft PR exposes the complete bootstrap for review.

Application scaffolding and feature implementation are explicitly outside this
design. They begin only after this written foundation is reviewed.
